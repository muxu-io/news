## Executive Summary
This week's updates include a new Yocto Project 6.0 LTS release and a point release for 5.3 (whinlatter). A potential issue with git cloning due to trailing slashes was identified and patched in the fetch2 git module. Several updates were made to meta-openembedded recipes to address dependencies and version upgrades. NVIDIA's Jetson ecosystem is seeing increased interest in the Thor platform and continued issue resolution for the Orin series.

## Critical/Urgent

*   **Yocto Project 6.0 Released:** The Yocto Project 6.0 (wrynose) is now available. This is a new LTS release, so evaluate its features and migration requirements. ([https://lists.yoctoproject.org/g/yocto-announce/message/386](https://lists.yoctoproject.org/g/yocto-announce/message/386))
*   **Yocto Project 5.3.4 Released:** A point release is available for the 5.3 (whinlatter) branch. ([https://lists.yoctoproject.org/g/yocto-announce/message/385](https://lists.yoctoproject.org/g/yocto-announce/message/385))
*   **Git Clone Issue:** A patch has been proposed to fix a trailing slash in the git clone command within the fetch2 module, which can cause issues with alternate object paths, especially with newer versions of git. This could affect source fetching in your builds. ([https://lists.yoctoproject.org/g/docs/message/9471](https://lists.yoctoproject.org/g/docs/message/9471), [https://lists.yoctoproject.org/g/docs/message/9470](https://lists.yoctoproject.org/g/docs/message/9470))

## Yocto Core

*   **sstate object validation:** TMPDIR paths should not exist within sstate objects. New checks have been added to detect and flag these problematic paths. This can improve build reproducibility and avoid unexpected side effects. ([https://github.com/openembedded/openembedded-core/commit/907af8fb448e2f9ecf8e0439f2d8c7c397fb873f](https://github.com/openembedded/openembedded-core/commit/907af8fb448e2f9ecf8e0439f2d8c7c397fb873f))
*   **pseudo Upgrade:** The pseudo recipe has been upgraded to version 1.9.7, bringing in fixes for rename/renameat/renameat2. ([https://github.com/openembedded/openembedded-core/commit/17567738711d525d9f2b85e54ace2048901e4c34](https://github.com/openembedded/openembedded-core/commit/17567738711d525d9f2b85e54ace2048901e4c34))
*   **expat Upgrade:** expat recipe updated to version 2.8.1, bringing in security fixes for CVE-2026-45186. This resolves a quadratic runtime from attribute name collision checks that can allow DoS attacks. ([https://github.com/openembedded/openembedded-core/commit/4d7dd423b62e1e3c9f434d8c178e9dcc6529f010](https://github.com/openembedded/openembedded-core/commit/4d7dd423b62e1e3c9f434d8c178e9dcc6529f010))
*   **python3 Upgrade:** python3 recipe updated to version 3.14.5, resolving multiple CVEs. ([https://github.com/openembedded/openembedded-core/commit/0df68073ebaecb968448dcf516b9085efadc6d29](https://github.com/openembedded/openembedded-core/commit/0df68073ebaecb968448dcf516b9085efadc6d29))
*   **glib-2.0-native: Mitigate native tool path relocation issues:** gdb helper scripts installed by this component are now skipped in native builds. The helper scripts were installed to a path that matches the installed binary path. Since we relocate native tools, this path isn't relocated correctly. This results in the build directory leaking into sstate objects and native sstate objects can have differing path contents as a result, which causes problems. Whilst we could relocate it, gdb would not be searching in these paths for helpers anyway so the tools aren't needed and we can simply remove them in the native case. ([https://github.com/openembedded/openembedded-core/commit/117ed64803dc2c8176125835b2bbcc08225f45da](https://github.com/openembedded/openembedded-core/commit/117ed64803dc2c8176125835b2bbcc08225f45da))
